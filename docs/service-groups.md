# Service Groups

Service groups add an independent auth subject that can manage many services. A service can belong to many service groups.

## Data Model

Service groups use `service_groups` and `service_group_members`.

```sql
CREATE TABLE IF NOT EXISTS service_groups (
  id BIGSERIAL PRIMARY KEY,
  service_group_name TEXT NOT NULL UNIQUE,
  service_group_url TEXT NOT NULL UNIQUE,
  authorization_code_hash TEXT NOT NULL,
  authorization_code_masked TEXT NOT NULL,
  access_token TEXT NOT NULL DEFAULT '',
  access_token_expires_at BIGINT,
  token_version BIGINT NOT NULL DEFAULT 0,
  created_at BIGINT NOT NULL DEFAULT EXTRACT(EPOCH FROM NOW())::BIGINT,
  updated_at BIGINT NOT NULL DEFAULT EXTRACT(EPOCH FROM NOW())::BIGINT
);

CREATE TABLE IF NOT EXISTS service_group_members (
  service_group_id BIGINT NOT NULL REFERENCES service_groups(id) ON DELETE CASCADE,
  service_id BIGINT NOT NULL REFERENCES services(id) ON DELETE CASCADE,
  created_at BIGINT NOT NULL DEFAULT EXTRACT(EPOCH FROM NOW())::BIGINT,
  PRIMARY KEY (service_group_id, service_id)
);
```

`service_group_url` is generated by the server as `service-group://{id}` and is read-only.

## Token Behavior

Service groups only expose access tokens. They do not expose public refresh tokens.

Service group JWTs use:

- `subjectType=serviceGroup`
- `serviceGroupName=<name>`
- `tokenVersion=<version>`

Service JWTs use `subjectType=service`. Parsing remains compatible with older service tokens that do not contain `subjectType`.

## Verification

Normal service verification stays unchanged:

```http
POST /api/auth/verify
Service-Name: billing
Access-Token: <serviceAccessJwt>
Origin: https://billing.example.com
```

Service group verification uses `Service-Name` for the group and `Target-Service-Name` for the target service:

```http
POST /api/auth/verify
Service-Name: core-group
Target-Service-Name: billing
Access-Token: <serviceGroupAccessJwt>
```

Service group verification skips `Origin` and `Referer`, checks that the target service belongs to the group, and applies the target service QPS/QPM limit.

## Latest Key Idempotency

`POST /api/service-groups/token/latest` accepts `serviceGroupName + authorizationCode`.

The repository locks the service group row with `SELECT ... FOR UPDATE`, checks token remaining lifetime inside the lock, and only refreshes when the current access token has one hour or less remaining.

This means concurrent boundary-time calls converge on one refreshed token and one incremented `token_version`.
